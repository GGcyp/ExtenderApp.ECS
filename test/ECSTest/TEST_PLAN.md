# 测试计划（canonical test checklist）

说明：此文件为项目的统一测试清单与规范。所有自动或手写的测试代码在生成/编写之前应首先读取并参考本文件，确保测试覆盖面、命名与运行方式一致。

## 目的
- 统一测试用例分类与优先级
- 提供性能测试的基线与可调参数
- 明确异常与边界行为的期望

## 约定
- 测试使用中文描述断言信息。
- 重要的性能基准应使用 `BenchmarkDotNet` 或相当工具进行可重复测量。
- 性能/压力测试应提供可配置参数（例如 N、是否启用 Unsafe 路径）。
- 所有测试应在主线程上下文执行；若支持跨线程访问，需增加并发测试。

## 测试分类与建议用例
1. 功能性测试
   - 单组件增删改查：Add/Remove/Set/Get，验证读写一致性与替换行为。
   - 多组件组合：使用 `ComponentMask`/Archetype 创建实体，测试按掩码创建后各组件的读写。
   - 重复/非法操作：重复添加同一组件、对不存在实体操作、对已移除组件读写等应有明确异常或幂等行为。

2. 边界测试
   - 掩码边界：空掩码、满掩码（支持的最大组件数）、试图设置超出索引的组件应抛出。
   - Chunk 边界：Capacity 边界读写、越界访问应抛异常、Swap/Copy 在边界条件下的正确性。
   - 最大组件大小：验证对超过 MaxElementSize 的结构有适当拒绝或日志提示。

3. 异常行为测试
   - 读取不存在的组件应抛异常或返回明确错误（以 API 约定为准）。
   - 为已释放/已销毁的实体操作应抛异常或安全忽略。

4. 并发/线程安全测试（如果支持）
   - 在非主线程调用受限 API 应抛异常（主线程检测）。
   - 并发创建/读写（若支持并发）需验证数据一致性与无竞态。

5. 性能测试
   - 批量创建（Create N entities）、批量写入（Set N*components）、批量读取（Get N*components）。
   - 建议基准参数：小规模（1k）、中等（10k）、大规模（100k，根据机器能力）。
   - 测量项：总耗时、每操作吞吐（ops/ms）、GC 分配（分配 bytes / Gen0/1/2 次数）、非托管内存使用。
   - 对比：安全版本（检查）与 Unsafe 热路径的性能差异。

6. 压力与稳定性测试
   - 长时间循环创建/销毁实体（例如持续 1M 次操作或循环几小时），监测内存增长与崩溃。
   - 随机化操作序列（添加/移除/读取/销毁）以发现边缘 bug。

## 运行与工具建议
- 普通功能测试使用单元测试框架（xUnit/NUnit），将断言与错误归档为 CI 报告。
- 性能基准使用 `BenchmarkDotNet` 并在 CI 或专用机器上运行，生成可比对的报告。
- 压力测试建议在隔离环境（非 CI）运行，记录日志并进行内存与资源监控。

## 测试元数据约定（建议）
- 每个测试文件顶部包含注释：测试目的、所需前置条件、是否为性能/压力测试、推荐运行时间。
- 性能测试应提供参数化入口，例如环境变量或配置文件来调整 N 与是否使用 Unsafe。

## 扩展与维护
- 若新增重要 API（例如新的组件生命周期或并发支持），请同步更新本文件并为新行为添加对应测试。
- 长期建议：将关键性能场景编码为 BenchmarkDotNet 基准并纳入定期回归测试。
